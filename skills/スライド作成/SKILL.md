以下の手順でスライド資料を作成してください。

引数: $ARGUMENTS
（例: `/スライド作成 LT用にAI勉強会の内容をまとめて` `/スライド作成 クライアント提案の施策サマリー5枚` `/スライド作成 スライド/提案/クライアント提案.pptx を改善して`）

---

## モード判定

`$ARGUMENTS` の内容から、以下のどちらのモードで実行するか判定する：

- **作成モード**: 新規スライドを作成する（デフォルト）
- **改善モード**: 既存スライドをレビュー・改善する

### 改善モードの判定条件（いずれかに該当）
- 既存の `.pptx` ファイルパスが指定されている
- 「改善」「レビュー」「評価」「見直し」「ブラッシュアップ」等のキーワードを含む
- 「直近作成したスライド」等、既存スライドへの参照がある

改善モードの場合は **ステップR0** へ、作成モードの場合は **ステップ0** へ進む。

---

デザインシステム・テーマ定義・スライドタイプ一覧・改善モードの詳細は [reference.md](reference.md) を参照してください。

# 作成モード

## ステップ0：デザインテーマの選択

まず、使用するデザインテーマをユーザーに確認してください。テーマの詳細・インポート方法は [reference.md](reference.md) を参照。

```
【デザインテーマを選んでください】

1. quest（デフォルト） — プロフェッショナルな提案資料向け（ティール+ゴールド）
2. green（LT・勉強会向け） — 社内勉強会 スタイル（グリーン+ライム）
3. internal（朝会・勉強会向け） — 社内テンプレート準拠（グレー+ブルー）
4. corporate（リテールメディア提案向け） — 社内提案FMT（ティール+ゴールド）

（番号またはテーマ名で回答。省略時は1）
```

## ステップ1：コンテンツの収集

### 1-1. 指定ファイルの読み込み
- 議事録やナレッジファイルが指定されている場合は読み込む
- `自分について.md` を読み、発表者情報・所属を把握

### 1-2. ナレッジの自動検索（常に実行）
テーマに関連するナレッジを自動検索してください：
- `${VAULT_PATH}/ナレッジ/` 以下を Grep でテーマのキーワード検索
- `${VAULT_PATH}/議事録/` からも関連する議事録を検索
- ヒットしたファイルを読み込み、スライドに活用できる情報を抽出

**検索で見つかった関連ナレッジをユーザーに提示：**
```
### 自動検索で見つかった関連情報
- [[{ナレッジファイル名}]] — {関連する部分の要約}
→ これらの情報もスライドに反映しますか？
```

## ステップ2：スライド構成の設計

収集した情報をもとに、スライドの構成案をユーザーに提示してください。

### ストーリー構成の原則（必須）

提案・企画系のスライドは、以下の「前提→課題→データ→解決策」フローに従うこと：

```
1. 前提・与件整理（クライアントの状況・目的）
2. 課題の特定（データに基づく問題提起）
3. 要因分析（なぜその課題が起きているか）
4. 解決策（課題に直結する施策）
5. 実行計画（シミュレーション・予算・スケジュール）
6. 期待効果（KPI・次のステップ）
```

**チェックポイント（構成案提示前に自己検証）：**
- 各セクション間が「なぜ？」「だから何？」で繋がるか
- 全体像（Who×What戦略概要）が詳細の前にあるか
- 1スライド1メッセージになっているか（ヘッダーが主張文か）

LT・勉強会系は上記に縛られず、自由な構成でOK。

### 構成案の提示

```
【スライド構成案】（全XX枚）
1. 表紙 — タイトル
2. アジェンダ — セクション一覧
3. ...
```

ユーザーの承認を得てから次に進んでください。

## ステップ3：スライド生成

`createDeck()` で統一APIを使いスライドを生成します。80種のスライドタイプ一覧は [reference.md](reference.md) を参照。

```javascript
import { createDeck, draw, layout } from '${VAULT_PATH}/.claude/slide-assets/index.mjs';
const deck = createDeck('green', 'タイトル');  // テーマIDを指定
```

### デザインルール（厳守）

1. **カラーパレット**: `COLORS` 辞書のみ使用。独自の色は追加しない
2. **フォント**: `FONT_*` 定数のみ使用
3. **レイアウトパターン**: テーマのスライドタイプを適切に使い分ける
4. **content_fn パターン**: 定型メソッドで表現しきれないレイアウトは `add_content_slide()` の `content_fn` コールバックで自由にシェイプを配置
5. **情報密度**: 1スライドに詰め込みすぎない。1スライド1メッセージを意識
6. **テキスト量**: 箇条書きは1項目2行以内。フォントサイズ10pt未満は極力避ける

### 数値制約ルール（厳守）

生成するスライドは以下の制約を満たすこと。違反がある場合は自動で修正する。

| 制約項目 | 上限値 | 理由 |
|----------|--------|------|
| ヘッダーメッセージ | 30文字 | 一目で読める長さ |
| 本文1行 | 40文字 | 可読性確保 |
| 1スライドの総行数 | 12行 | 情報過密防止 |
| 1スライドの総文字数 | 400文字 | 聞き手の認知負荷上限 |
| 箇条書き項目数 | 5項目 | 5項目超は2列化 or 分割 |
| テーブル | 1個/スライド | 複数テーブルは分割 |
| リスト階層 | 最大2階層 | 深いネストは避ける |
| フォントサイズ下限 | 10pt | 可読性の最低ライン |

**制約違反時の自動対応:**
- 行数超過 → スライドを2枚に分割
- 箇条書き5項目超 → 2列グリッドに変換（`add_grid_slide`活用）
- 文字数超過 → 表現を簡潔化して再生成
- フォントサイズ不足 → `FONT_SIZE["caption"]`(10pt)以上に引き上げ

### コードの書き方

- 全スライドを **1つのNode.jsスクリプト (.mjs)** で生成する（PptxGenJS使用）
- `node script.mjs` で実行
- エラーが出たらデバッグして再実行する
- 最終的に動作確認が取れたコードで保存する

## ステップ3.5：視覚検証（自動）

生成したPPTXを画像に変換し、レイアウト崩れを自動検出・修正する。

### 3.5-1. スライド画像の生成

```bash
TMPDIR=$(mktemp -d)
soffice --headless --convert-to pdf --outdir "$TMPDIR" "生成したファイル.pptx"

python3 -c "
from pdf2image import convert_from_path
import os
tmpdir = '$TMPDIR'
pdf_path = [f for f in os.listdir(tmpdir) if f.endswith('.pdf')][0]
images = convert_from_path(os.path.join(tmpdir, pdf_path), dpi=150)
for i, img in enumerate(images, 1):
    img.save(os.path.join(tmpdir, f'slide_{i}.png'))
print(f'{len(images)} slides exported to {tmpdir}')
"
```

**フォールバック**: LibreOffice未インストールの場合は `brew install --cask libreoffice` を提案。`pdf2image` がない場合は `pip install pdf2image` を実行。

### 3.5-2. 画像の目視チェック（Claudeが実行）

生成したPNG画像を Read ツールで読み込み、以下の観点で自動チェック:

- テキストのはみ出し・切れ
- 要素の重なり
- 余白の不均一
- フォントサイズの視認性
- カラーパレットの統一性

### 3.5-3. 問題の自動修正

問題が見つかった場合、PptxGenJSスクリプトを修正して再生成し、再度画像生成→チェック。
**最大2ラウンド**で打ち切り、残りの問題はユーザーに報告。

**よくある修正パターン:**
- 下部切れ → マージン縮小 or スライド分割
- 5項目超の箇条書き → 2列グリッド化
- テキストオーバーフロー → フォントサイズ調整 or 文言簡潔化

## ステップ4：保存と報告

### カテゴリ判定

| カテゴリ | フォルダ | 判定基準 |
|---------|---------|---------|
| 勉強会・ハンズオン | `スライド/勉強会/` | ハンズオン、ワークショップ、勉強会向け資料 |
| 提案・企画 | `スライド/提案/` | 社内提案、企画書、導入提案 |
| ガイド・マニュアル | `スライド/ガイド/` | セットアップガイド、完全ガイド、手順書 |
| LT・発表 | `スライド/LT・発表/` | LT会、朝会発表、カンファレンス発表 |
| 日報 | `スライド/日報/` | 日報スライド、日報サマリー |

保存先: `スライド/{カテゴリフォルダ}/{スライドタイトル}.pptx`

完成報告として保存先パス・スライド枚数・各スライドの概要を出力。

# 改善モード

7人のレビュー専門エージェントが並列で評価し、全スコアが満点（5/5）になるまで改善ループを回す。
詳細な評価基準・ループ制御ルールは [reference.md](reference.md) を参照。

## 概要フロー

1. **R0**: 対象スライド特定 → テキスト抽出 + スライド画像生成
2. **R1**: 7人のレビュアーを並列起動（6人テキスト評価 + 1人ビジュアル評価）
3. **R2**: 評価結果集約・提示（7軸×5点 = 35点満点）
4. **R3**: 5点未満の軸を修正 → 再評価ループ（最大3ラウンド）
5. **R4**: 全軸5/5で完了報告
